## 1. Жалпы мәліметтер

### 1.1 Жобаның толық атауы

**Қазақша:** Деректерді шифрлау арқылы жеке қаржыны есепке алуға арналған қауіпсіз мобильді қосымша әзірлеу.

**Орысша:** Разработка мобильного приложения для безопасного учёта личных финансов с шифрованием данных.

**Ағылшынша:** Development of a mobile application for secure personal finance management with data encryption.

### 1.2 Жобаның қысқаша атауы

`SecureFinance KZ` — Android операциялық жүйесіне арналған жеке қаржыны есепке алатын, барлық деректерді соңғыдан-соңғыға (end-to-end) шифрлайтын мобильді қосымша.

### 1.3 Тапсырыс беруші

Дипломдық жоба ретінде әзірленеді. Тапсырыс беруші — жоғары оқу орны (университет) және ғылыми жетекші.

### 1.4 Әзірлеуші

Информатика/Ақпараттық жүйелер/Бағдарламалық инженерия мамандығының бітіруші студенті.

### 1.5 Жобаны әзірлеу мерзімі

Күнтізбелік жоспарға сәйкес (әдетте 4–6 ай). Нақты мерзімдер 14-бөлімде көрсетілген.

### 1.6 Жоба нәтижесі

1. Android операциялық жүйесіне арналған APK файлы.
2. Жобаның бастапқы коды (Git-репозиторий).
3. Техникалық құжаттама (осы ТЗ, архитектура сипаттамасы, API құжаттамасы).
4. Пайдаланушы нұсқаулығы.
5. Дипломдық түсініктеме жазба (пояснительная записка).

## 2. Жобаны әзірлеу негіздемесі

### 2.1 Өзектілігі

Цифрлық экономиканың дамуымен бірге жеке қаржыны есепке алу үшін мобильді қосымшаларды пайдалану күн санап кеңейіп келеді. Қазақстан Республикасында смартфондарды пайдалану деңгейі 85%-дан асты, ал халықтың едәуір бөлігі күнделікті шығындарын қағаз кітапшамен емес, электронды құралдар арқылы бақылайды.

Дегенмен, қолданыстағы қаржылық қосымшалардың басым бөлігі пайдаланушы деректерін бұлтқа (cloud) жүктейді, ал бұл мынадай тәуекелдерге алып келеді:

- жеке қаржылық ақпараттың үшінші тұлғаларға (жарнама желілерінде, дата-брокерлерге) берілуі;
- серверлерге кибершабуыл жасалғанда деректердің ағып кетуі;
- қосымшаны орнатқан телефон жоғалған/ұрланған жағдайда құпия ақпаратқа заңсыз қол жеткізу;
- Қазақстан Республикасының «Дербес деректер және оларды қорғау туралы» Заңының талаптарын толық орындамау.

Осы мәселелерді шешу үшін **барлық сезгіш деректерді тек құрылғыда (local-first), криптографиялық жолмен шифрлап сақтайтын** мобильді қосымшаны әзірлеу өзекті мәселе болып табылады.

### 2.2 Проблемалық жағдай

Қолданыстағы жеке қаржы қосымшаларының негізгі кемшіліктері:

1. **Cloud-first тәсіл.** Деректер серверде сақталып, шифрлау тек транспорт деңгейінде (TLS) ғана жүзеге асырылады. Сервер администраторы, не болмаса құзырлы орган ашық мәтінмен деректерге қол жеткізе алады.
2. **Weak-шифрлау немесе мүлде шифрламау.** SQLite-тың стандартты нұсқасы шифрланбаған мәтін ретінде сақталады (жай `root`-құрылғыдан файл көшіріп алып, мазмұнын оқуға болады).
3. **PIN-ді ашық немесе әлсіз хэшпен сақтау** (MD5, SHA-1 итерациясыз).
4. **Биометрияны жалған тіреуішпен іске асыру** (тек UI кідіртетін, нақты криптографиялық бағасы жоқ).
5. **Қазақ тілінің болмауы** — қолданыстағы танымал қосымшалар (Monefy, Wallet, CoinKeeper) негізінен ағылшын/орыс тілдерінде.

### 2.3 Дипломдық жобаның ғылыми-тәжірибелік құндылығы

- Android Keystore, PBKDF2, AES-GCM, SQLCipher технологияларын бірыңғай жүйеге біріктірудің практикалық мысалы.
- Қазақ тілді қаржылық қосымшаның отандық пайдаланушыларға арналған моделі.
- Жергілікті сақтау + соңғыдан-соңғыға шифрлау архитектурасының толыққанды жүзеге асырылуы.
- Қазақстан Республикасының «Дербес деректер және оларды қорғау туралы» Заңына сәйкес келетін техникалық шешім үлгісі.

## 3. Жобаның мақсаты мен міндеттері

### 3.1 Мақсаты

Android платформасында жұмыс істейтін, пайдаланушының жеке қаржылық деректерін (табыстар, шығындар, шоттар, категориялар) криптографиялық тұрғыдан қорғалған түрде жергілікті сақтайтын, қолдануға ыңғайлы және Қазақстан пайдаланушыларына бейімделген мобильді қосымшаны әзірлеу.

### 3.2 Міндеттері

1. Пән саласын зерттеу, қолданыстағы аналогтарды талдау, олардың қауіпсіздік кемшіліктерін анықтау.
2. Қосымшаның функционалдық және функционалдық емес талаптарын анықтау, техникалық тапсырманы дайындау.
3. Қосымшаның архитектурасын (MVVM + Clean Architecture элементтері) жобалау.
4. Криптографиялық қабатты (key derivation, encryption, key storage) жобалау және іске асыру.
5. Android Keystore, EncryptedSharedPreferences, SQLCipher технологияларын интеграциялау.
6. Деректер базасының схемасын (Room + SQLCipher) жобалау және жүзеге асыру.
7. Пайдаланушы интерфейсін Material Design 3 негізінде, қазақ тілінде әзірлеу.
8. Биометриялық аутентификация (BiometricPrompt) модулін енгізу.
9. Автоматты құлыптау, FLAG_SECURE, экспорт/импорт функцияларын іске асыру.
10. Қосымшаны бірлік (unit), интеграциялық және қауіпсіздік тестілерімен тексеру.
11. Техникалық құжаттама және пайдаланушы нұсқаулығын әзірлеу.
12. Жобаны қорғауға дайындау (презентация, демо-видео).

### 5.3 Валютаға қойылатын талаптар

- Негізгі валюта — **қазақстандық теңге (KZT, ₸)**.
- Қосымша валюталарды (USD, EUR, RUB) таңдау мүмкіндігі.
- Валюталар арасындағы бағам — қолмен енгізу (офлайн режимде).

## 6. Функционалдық талаптар

### 6.1 Пайдаланушы рөлдері

Қосымша жеке пайдалануға арналғандықтан, бір ғана рөл қарастырылады — **Пайдаланушы (User)**. Сыртқы серверлер, әкімшілік панель немесе көпрөлді жүйе жоқ.

### 6.2 Функционалдық блоктар

Қосымша келесі функционалдық блоктардан тұрады:

- **FB-1:** Аутентификация және қауіпсіздік
- **FB-2:** Шоттарды басқару (Accounts)
- **FB-3:** Категорияларды басқару (Categories)
- **FB-4:** Транзакцияларды басқару (Transactions)
- **FB-5:** Статистика және талдау (Statistics)
- **FB-6:** Деректерді экспорттау/импорттау
- **FB-7:** Баптаулар (Settings)

### 6.3 FB-1. Аутентификация және қауіпсіздік

#### FR-1.1 Алғашқы орнату (Setup)

- Қосымшаны **алғашқы рет** қосқанда пайдаланушыдан 4–8 таңбалы PIN-код енгізу сұралады.
- PIN қайтадан расталады (қайталап енгізу).
- Жүйе PIN-тен PBKDF2 арқылы кілт туғызады, кездейсоқ 32 байт DB-кілтін генерациялайды және оны шифрлап сақтайды (8.2 бөлімді қараңыз).
- PIN ашық мәтінмен ешқайда сақталмайды.

#### FR-1.2 Кіру (Unlock)

- Қосымшаны қайта қосқанда **аутентификация экраны** көрсетіледі.
- Пайдаланушы PIN енгізеді немесе (қосулы болса) биометрияны пайдаланады.
- 5 рет қате енгізгенде 30 секундтік кешіктіру іске қосылады. Әр қате әрекеттен кейін кешіктіру уақыты экспоненциалды өседі (30с → 60с → 2мин → 5мин → 10мин).
- 10 рет қате әрекеттен кейін пайдаланушыға ескерту беріледі: «Қайта орнату ғана деректерді қалпына келтірмейді, бірақ қосымшаны қайта бастайды».

#### FR-1.3 Биометриялық аутентификация

- Егер құрылғыда саусақ ізі/бет-сканер бар болса, пайдаланушы **баптаулардан** биометрияны қоса алады.
- Биометрия `BiometricPrompt` + `CryptoObject` арқылы Android Keystore-да сақталған кілтті ашу үшін қолданылады.
- Биометрияны ауыстыру/жою кезінде Keystore кілті автоматты түрде жарамсыз болады (invalidated); осы жағдайда PIN міндетті болып қайта жанданады.

#### FR-1.4 Автоматты құлыптау

- Қосымша **фонға шыққанда** автоматты түрде таймер іске қосылады.
- Әдепкі уақыт — **60 секунд** (баптауларда 30с, 1мин, 3мин, 5мин, ешқашан — таңдауға болады).
- Белгіленген уақыт өткен соң сессия жабылады: DB-кілт жадтан өшіріледі, қосымша құлыпталады.
- Қосымшаны қайта ашқанда міндетті түрде аутентификация өтілуі керек.

#### FR-1.5 PIN-ді ауыстыру

- Пайдаланушы баптаулардан PIN-ді ауыстыра алады.
- Ескі PIN тексеріледі, содан кейін жаңа PIN енгізіліп, расталады.
- Жаңа PIN арқылы DB-кілт қайта шифрланып сақталады (DB өзі қайта шифрлануы қажет емес).

#### FR-1.6 Скриншот және Recents-экранды қорғау

- Барлық сезгіш экрандарда `FLAG_SECURE` орнатылады.
- Бұл скриншот жасауға, экранды жазуға, recents-экранда мазмұнды көрсетуге тосқауыл қояды.

#### FR-1.7 Backup-тан қорғау

- `android:allowBackup="false"` және `dataExtractionRules` арқылы autobackup мен device-transfer механизмдері өшіріледі.
- Бұл пайдаланушы деректерінің Google Cloud-қа немесе басқа құрылғыға ашық мәтінмен көшірілуіне жол бермейді.

### 6.4 FB-2. Шоттарды басқару

#### FR-2.1 Шот жасау

Пайдаланушы жаңа шот қоса алады. Шот өрістері:
- Атауы (мысалы: «Каспи қызыл», «Қолма-қол», «Халық жинақ»)
- Түрі: Қолма-қол / Карта / Жинақ шоты / Басқа
- Бастапқы баланс
- Валюта (KZT, USD, EUR, RUB)
- Түс (визуалды бөлу үшін)

#### FR-2.2 Шотты өңдеу / жою

- Шоттың атын, түрін, валютасын, түсін өзгертуге болады.
- Шотты жойғанда, оған байланысты барлық транзакциялар каскадты (CASCADE) түрде жойылады.
- Жою алдында растау диалогы шығарылады.

#### FR-2.3 Шоттардың тізімін көру

- Барлық шоттар тізім түрінде көрсетіледі.
- Әр шот бойынша ағымдағы балансы, валютасы, соңғы транзакциясының күні көрсетіледі.
- Шоттардың жалпы балансы автоматты түрде есептеледі.

### 6.5 FB-3. Категорияларды басқару

#### FR-3.1 Әдепкі категориялар

Алғашқы орнатуда келесі категориялар автоматты түрде жасалады:

**Шығыс категориялары:** Азық-түлік, Көлік, Тұрғын үй, Байланыс, Киім, Денсаулық, Ойын-сауық, Білім, Мейрамхана/кафе, Сыйлықтар, Салықтар/төлемдер, Басқа шығыстар.

**Кіріс категориялары:** Жалақы, Сыйлық, Қосымша жұмыс, Зейнетақы/жәрдемақы, Инвестиция, Қарыз қайтарылды, Басқа кірістер.

#### FR-3.2 Категория қосу / өңдеу / жою

- Пайдаланушы жеке категорияларын қоса алады (аты, түсі, түрі — кіріс/шығыс).
- Категорияны жойғанда, оған байланысты транзакциялардың `categoryId` өрісі `NULL`-ге орнатылады (транзакциялардың өзі жойылмайды).

### 6.6 FB-4. Транзакцияларды басқару

#### FR-4.1 Транзакция қосу

Жаңа транзакция (кіріс немесе шығыс) қосу. Өрістер:
- Түрі: Кіріс / Шығыс
- Сомасы (оң сан)
- Шоты (тізімнен таңдау)
- Категориясы (транзакция түріне сәйкес тізімнен таңдау)
- Күні мен уақыты (әдепкі — қазіргі уақыт)
- Ескерту (опционалды мәтін)

Транзакция сақталған кезде байланысты шоттың балансы автоматты түрде өзгереді (кіріс → баланс өседі, шығыс → баланс азаяды). Бұл операция **транзакциялық** (бір Room транзакция ішінде) орындалуы керек.

#### FR-4.2 Транзакцияны өңдеу

- Пайдаланушы кез келген өрісті өзгерте алады.
- Өңдеу кезінде шоттың балансы дұрыс қайта есептелуі керек (ескі мәнді қайтару + жаңа мәнді қолдану).

#### FR-4.3 Транзакцияны жою

- Растау диалогы шығарылады.
- Жойылған транзакцияның сомасы шоттан автоматты түрде қайтарылады.

#### FR-4.4 Транзакциялар тізімі

- Барлық транзакциялар күні бойынша кему тәртібінде (жаңасынан ескіге) көрсетіледі.
- Тізім күндер бойынша топтастырылады (мыс.: «Бүгін», «Кеше», «12 қараша»).

#### FR-4.5 Сүзгі (фильтр) және іздеу

Транзакцияларды сүзуге болады:
- Кезеңі бойынша (бүгін, осы апта, осы ай, осы жыл, арнайы аралық).
- Түрі бойынша (кіріс / шығыс / бәрі).
- Шоты бойынша.
- Категориясы бойынша.
- Ескертудегі мәтін бойынша іздеу.

### 6.7 FB-5. Статистика және талдау

#### FR-5.1 Басты бет (Dashboard)

Басты бетте көрсетіледі:
- Барлық шоттардың жалпы балансы (негізгі валютаға (KZT) шоғырландырылған).
- Ағымдағы ай: жалпы кіріс, жалпы шығыс, айырмасы.
- Соңғы 10 транзакция.
- Ай бойынша шығындардың категориялар бойынша бөлінуі (кіші pie-chart).

#### FR-5.2 Толық статистика экраны

- Кезеңді таңдау мүмкіндігі (апта / ай / тоқсан / жыл / арнайы аралық).
- Кіріс пен шығыстың арасындағы балансы (bar chart).
- Категориялар бойынша шығындардың үлесі (pie chart).
- Күнделікті/апта сайынғы динамика (line chart).
- Ең көп шығындалған категориялардың топ-5 тізімі.

#### FR-5.3 Графиктер

Графиктерді құру үшін **MPAndroidChart** кітапханасы қолданылады.

### 6.8 FB-6. Деректерді экспорттау/импорттау

#### FR-6.1 Шифрланған backup жасау

- Пайдаланушы барлық деректерін бір файлға экспорттай алады.
- Файл форматы: `.sfkz` (меншікті кеңейтім, JSON + AES-GCM).
- Экспорттау кезінде пайдаланушы **экспорт құпиясөзін** енгізеді (ағымдағы PIN-нен бөлек болуы мүмкін).
- Құпиясөзден PBKDF2 арқылы кілт шығарылып, файлдың мазмұны AES-256-GCM арқылы шифрланады.

#### FR-6.2 Шифрланған backup-тан қалпына келтіру

- Пайдаланушы `.sfkz` файлын таңдайды, экспорт кезіндегі құпиясөзді енгізеді.
- Деректер ағымдағы дерекқорға импортталады (қайта жазу немесе біріктіру — таңдауға болады).

#### FR-6.3 CSV экспорт (опция)

- Қарапайым CSV файлына экспорттау (шифрланбаған).
- Пайдаланушыға бұл файлдың **шифрланбайтынын** ескерту беріледі.

### 6.9 FB-7. Баптаулар

#### FR-7.1 Қауіпсіздік баптаулары

- PIN-ді өзгерту.
- Биометрияны қосу/өшіру.
- Автоқұлыптау уақытын таңдау.
- `FLAG_SECURE`-ді мәжбүрлеп қосу/өшіру (advanced).

#### FR-7.2 Интерфейс баптаулары

- Тілді таңдау (қазақша / орысша).
- Тақырып: ашық / қараңғы / жүйелік.
- Негізгі валютаны таңдау.

#### FR-7.3 Деректерді басқару

- Backup жасау.
- Backup-тан қалпына келтіру.
- **Барлық деректерді өшіру (Factory Reset)** — растау диалогынан кейін барлық дерекқор мен кілттер толық жойылады.

#### FR-7.4 Ақпарат

- Қосымша туралы.
- Құпиялылық саясаты.
- Пайдаланушы келісімі.
- Құжаттама/нұсқаулық.

- Минимум қаріп өлшемі — 14sp.
- Визуалды элементтердің контрасты WCAG AA талаптарына сай (4.5:1).

### 7.4 Қолжетімділік (Accessibility)

- Барлық интерактивті элементтердің `contentDescription` көрсетілген.

### 7.5 Тасымалданғыштық

## 8. Қауіпсіздік жүйесіне қойылатын талаптар

**Бұл бөлім диплом жобасының басты инновациялық бөлігі болып табылады.**

### 8.1 Қауіптерді модельдеу (Threat Model)

Қосымша келесі қауіптерден қорғанысы керек:

| № | Қауіп | Қауіп-қатердің түрі | Қорғаныс механизмі |
|---|---|---|---|
| T-1 | Ұрланған/жоғалған құрылғыдан деректерді оқу | Құпиялылық | PIN + SQLCipher + AES-256 |
| T-2 | Root-құрылғыдан `/data/data/` ішінен DB файлын көшіру | Құпиялылық | SQLCipher (файл-деңгей шифрлау) |
| T-3 | PIN-ді брут-форс арқылы табу | Аутентификация | PBKDF2 100к итерация + экспоненциалды кешіктіру |
| T-4 | Жадтан (RAM dump) кілт алу | Құпиялылық | Кілтті `ByteArray` + `fill(0)` тазалау, әдепкі Kotlin `String`-ті қолданбау |
| T-5 | Биометрия ауыстырылғанда ескі кілтке қолжетімдік | Аутентификация | Keystore `setInvalidatedByBiometricEnrollment(true)` |
| T-6 | Recents экранда деректердің көрінуі | Құпиялылық | `FLAG_SECURE` |
| T-7 | Android autobackup арқылы Google Cloud-қа деректер ағуы | Құпиялылық | `allowBackup=false`, `dataExtractionRules` |
| T-8 | Экспорт файлының ашық мәтінмен сақталуы | Құпиялылық | `.sfkz` — AES-GCM бөлек құпиясөзбен |
| T-9 | Clipboard арқылы сезгіш деректердің ағуы | Құпиялылық | PIN өрістері `FLAG_SECURE` + copy/paste шектеуі |
| T-10 | Қате енгізулер арқылы дерекқорды бүлдіру | Тұтастық | Room @Transaction + валидация |

### 8.2 Кілт басқару схемасы

Қосымша **иерархиялық кілт сызбасын (key hierarchy)** пайдаланады:

```
┌─────────────────────────────────────────────────────┐
│  1. Пайдаланушы PIN (4–8 таңба, ашық мәтінде)       │
└────────────────────────┬────────────────────────────┘
                         │ PBKDF2-SHA256
                         │ salt: 16 байт (кездейсоқ)
                         │ iterations: 100,000
                         │ keyLen: 256 бит
                         ▼
┌─────────────────────────────────────────────────────┐
│  2. K_user (256 бит, жадта ғана)                    │
└────────────────────────┬────────────────────────────┘
                         │ AES-256-GCM шифрлау
                         │ + 12 байт кездейсоқ IV
                         ▼
┌─────────────────────────────────────────────────────┐
│  3. Enc(K_db) — EncryptedSharedPreferences          │
│     (Android Keystore master-кілтпен қосымша қабат) │
└────────────────────────┬────────────────────────────┘
                         │ AES-256-GCM шешу
                         ▼
┌─────────────────────────────────────────────────────┐
│  4. K_db (256 бит, кездейсоқ, жадта ғана)           │
└────────────────────────┬────────────────────────────┘
                         │ SQLCipher
                         │ (AES-256-CBC + HMAC-SHA512,
                         │  PBKDF2 64000 итерация)
                         ▼
┌─────────────────────────────────────────────────────┐
│  5. Шифрланған SQLite файлы                         │
│     /data/data/kz.diploma.../databases/*.db         │
└─────────────────────────────────────────────────────┘
```

**Негізгі қағидаттар:**
- PIN ешқашан сақталмайды (тек хэші — тексеру үшін).
- `K_user` PIN енгізілген кезде ғана жадта болады, операция аяқталғаннан кейін `fill(0)` арқылы өшіріледі.
- `K_db` сессия ішінде ғана жадта болады, қосымша фонға шыққанда өшіріледі (8.5-ті қараңыз).
- Жоғарғы деңгейдегі мастер-кілт Android Keystore-да (TEE/StrongBox) сақталады.

### 8.3 Қолданылатын криптографиялық алгоритмдер

| Функция | Алгоритм | Параметрлер |
|---|---|---|
| Кілт деривациясы | PBKDF2-HMAC-SHA256 | 100 000 итерация, 256 бит кілт |
| Симметриялық шифрлау | AES | 256 бит, GCM режимі, 96-бит IV, 128-бит tag |
| Хэштеу (PIN тексеру) | SHA-256 (PBKDF2 негізінде) | Уақыт-тұрақты салыстыру |
| Деректер базасын шифрлау | SQLCipher 4.x | AES-256-CBC + HMAC-SHA512 |
| Backup файлдарын шифрлау | AES-256-GCM | PBKDF2-тен шыққан кілт |
| Кездейсоқ сандар | `SecureRandom` | SHA1PRNG немесе жүйелік |

**Ескерту:** MD5, SHA-1, DES, 3DES, RC4, ECB-режимі **қолданылмайды**.

### 8.4 Android Keystore интеграциясы

- `EncryptedSharedPreferences` ішкі мастер-кілтті `MasterKey.KeyScheme.AES256_GCM` арқылы Keystore-да сақтайды.
- Қолжетімді болса, `StrongBox` (арнайы қауіпсіздік чипі) қолданылады.
- Биометриялық кілт мынадай сипаттамалармен жасалады:
  - `setUserAuthenticationRequired(true)`
  - `setInvalidatedByBiometricEnrollment(true)`
  - `setUserAuthenticationParameters(0, AUTH_BIOMETRIC_STRONG)`

### 8.5 Сессияны басқару

- Қосымша іске қосылғанда `SessionManager.dbKey = null`.
- PIN/биометрия сәтті өткеннен кейін `dbKey` жадқа қойылады.
- Әр пайдаланушы әрекетінде (`onUserInteraction`) `lastActive` жаңартылады.
- `onPause`/`onStop` шақырылған сәттен бастап таймер іске қосылады.
- Таймер таусылғанда `dbKey.fill(0)` + DB-байланыс жабылады + құлыптау экраны ашылады.

### 8.6 Қосымша қауіпсіздік шаралары

- **ProGuard/R8:** шығарылым (release) құрастыруында міндетті түрде кодты абфусцирлеу.
- **Debug тексерісі:** release-де `BuildConfig.DEBUG` болса да, стек-трейс логтары тазартылады (`Timber` Release-tree).
- **Network Security Config:** clear-text трафик толық тыйылған (`cleartextTrafficPermitted=false`).
- **Root-тексеру (опционалды):** root-құрылғыда жұмыс істеуге ескерту.
- **SafetyNet / Play Integrity (болашаққа):** қосымшаның шынайылығын серверден тексеру (егер болашақта cloud-синхронизация қосылса).

## 9. Жүйенің архитектурасы

### 9.1 Архитектуралық үлгі

Қосымша **MVVM (Model–View–ViewModel)** үлгісін Clean Architecture элементтерімен біріктіріп қолданады.

```
┌───────────────────────────────────────────────────────┐
│                  PRESENTATION LAYER                   │
│   Activities / Fragments / Adapters / Layouts (XML)   │
└───────────────────┬───────────────────────────────────┘
                    │ observes LiveData / StateFlow
                    ▼
┌───────────────────────────────────────────────────────┐
│                   VIEWMODEL LAYER                     │
│   AuthViewModel, DashboardVM, TransactionVM, ...      │
└───────────────────┬───────────────────────────────────┘
                    │ invokes
                    ▼
┌───────────────────────────────────────────────────────┐
│                   REPOSITORY LAYER                    │
│         FinanceRepository (single source)             │
└───┬─────────────────────────────────────┬─────────────┘
    │                                     │
    ▼                                     ▼
┌─────────────────────┐         ┌─────────────────────┐
│   DATA LAYER        │         │  SECURITY LAYER     │
│   Room + SQLCipher  │         │  CryptoManager,     │
│   DAO, Entities     │         │  SessionManager,    │
│                     │         │  BiometricHelper    │
└─────────────────────┘         └─────────────────────┘
```

| `ui.dashboard` | Басты бет |


## 10. Деректер базасының құрылымы

### 10.1 Кестелер

#### Кесте `accounts`

| Өріс | Түрі | Шектеулер | Сипаттамасы |
|---|---|---|---|
| `id` | INTEGER | PRIMARY KEY, AUTOINCREMENT | Бірегей идентификатор |
| `name` | TEXT | NOT NULL | Шоттың аты |
| `type` | TEXT | NOT NULL | CASH / CARD / SAVINGS / OTHER |
| `balance` | REAL | NOT NULL DEFAULT 0 | Ағымдағы баланс |
| `currency` | TEXT | NOT NULL DEFAULT 'KZT' | Валюта коды |
| `color` | INTEGER | NOT NULL | ARGB түс |
| `createdAt` | INTEGER | NOT NULL | Unix timestamp (мс) |

#### Кесте `categories`

| Өріс | Түрі | Шектеулер | Сипаттамасы |
|---|---|---|---|
| `id` | INTEGER | PRIMARY KEY, AUTOINCREMENT | Бірегей идентификатор |
| `name` | TEXT | NOT NULL | Категория аты |
| `type` | TEXT | NOT NULL | INCOME / EXPENSE |
| `color` | INTEGER | NOT NULL | ARGB түс |

#### Кесте `transactions`

| Өріс | Түрі | Шектеулер | Сипаттамасы |
|---|---|---|---|
| `id` | INTEGER | PRIMARY KEY, AUTOINCREMENT | Бірегей идентификатор |
| `amount` | REAL | NOT NULL | Сома (оң сан) |
| `type` | TEXT | NOT NULL | INCOME / EXPENSE |
| `accountId` | INTEGER | FK → accounts.id, CASCADE | Байланысты шот |
| `categoryId` | INTEGER | FK → categories.id, SET NULL | Категория |
| `note` | TEXT | NOT NULL DEFAULT '' | Ескерту |
| `date` | INTEGER | NOT NULL | Транзакция уақыты |

**Индекстер:** `transactions(accountId)`, `transactions(categoryId)`, `transactions(date)`.

### 10.2 Негізгі SQL сұранымдар (концепциялық)

```sql
-- Жалпы баланс
SELECT IFNULL(SUM(balance), 0) FROM accounts;

-- Айлық кіріс
SELECT IFNULL(SUM(amount), 0) FROM transactions
WHERE type = 'INCOME' AND date BETWEEN :monthStart AND :now;

-- Категориялар бойынша шығындар
SELECT c.name, SUM(t.amount) AS total
FROM transactions t LEFT JOIN categories c ON t.categoryId = c.id
WHERE t.type = 'EXPENSE' AND t.date BETWEEN :from AND :to
GROUP BY t.categoryId ORDER BY total DESC;
```

### 10.3 Миграциялар

Алғашқы нұсқа `version = 1`. Кейінгі нұсқаларда Room Migration классы арқылы миграциялар жазылады. Диплом жобасында `fallbackToDestructiveMigration()` қолдануға рұқсат.

## 13. Тестілеу жоспары

### 13.1 Бірлік (Unit) тестілеу

Тексерілетін компоненттер:

- `CryptoManager`:
  - PBKDF2 детерминирленген шығыс (бір PIN + бір salt = бір нәтиже).
  - Дұрыс PIN → сәтті ашу; қате PIN → `null`.
  - AES-GCM шифр + дешифр қайтарымдылығы.
  - PIN ауыстыру: ескі DB-кілт сақталады.
- `SessionManager`:
  - `shouldLock()` уақыт біткенде `true`.
  - `lock()` кілтті тазалайды.
- Repository:
  - `addTransaction()` баланс дұрыс өзгертеді (income → +, expense → –).
  - Транзакция жойылғанда баланс қалпына келеді.
- Formatters (valuta, күні).

Мақсатты жабу: **≥ 70%** негізгі бизнес-логика.

### 13.2 Интеграциялық тестілеу

- Room + SQLCipher: шифрланған DB құру, жазу, оқу.
- Көп транзакциялы операциялар (`@Transaction`).
- EncryptedSharedPreferences интеграциясы.

### 13.3 UI (Espresso) тестілеу

Кем дегенде мына сценарийлер:

1. Алғашқы ашу → Setup экраны ашылады.
2. PIN орнату → Dashboard ашылады.
3. Қайта ашу → Auth экраны → PIN-мен кіру → Dashboard.
4. Транзакция қосу → тізімде пайда болады, баланс өзгереді.
5. Backup жасау → файл пайда болады, қалпына келтіруге болады.

### 13.4 Қауіпсіздік тестілеу

- Дерекқор файлын **тікелей** оқу (adb pull) — шифрланған болғаны расталады.
- 100 рет қате PIN — жүйе бұғатталады.
- Биометрия ауыстырылғанда — кілт жарамсыз болады.
- `FLAG_SECURE` — скриншот алуға болмайтыны.
- Android `StrictMode` VmPolicy — disk-encryption тексеру.

### 13.5 Өнімділік тестілеу

- 10 000 транзакциямен статистика < 3с.
- Cold-start < 2с.
- PBKDF2 100к итерация < 1с (орта құрылғы).

### 13.6 Қолдық тестілеу (Manual)

- Әртүрлі құрылғылар: Samsung, Xiaomi, Google Pixel.
- Android 7, 9, 11, 13, 14 — кемінде 3 нұсқаны тексеру.
